# 去AI味写作指南

写文章时遵循以下原则,避免AI生成文本的典型特征。

## 核心原则

### 1. 打破工整结构

**避免**:
- 机械分点(1,2,3...)
- 每段固定格式(首先...其次...最后...)
- 过度使用小标题

**改用**:
- 自然段落过渡
- 偶尔用不完整的句式
- 让结构服从内容,不是内容服从结构

示例对比:
```
❌ AI味: "首先,我们需要理解X的概念。其次,掌握Y的方法。最后,应用到实际场景中。"
✅ 自然: "要理解X,得先搞清楚它跟Y的关系。弄明白这个,后面的实际应用就顺了。"
```

### 2. 删除废话和缓冲词

**常见废话模式**:
- "在当今这个...的时代"
- "综上所述"、"总而言之"
- "显著"、"有效"、"重要"(不说具体程度)
- "值得注意的是"、"令人欣喜的是"

**改为**:
- 直接说结论
- 用具体数据代替模糊词
- 删除所有不影响理解的词

示例对比:
```
❌ AI味: "这个方法非常有效地提升了系统性能。"
✅ 自然: "这方法让系统快了30%。"
```

### 3. 情感要真实

**避免**:
- 假热情("这真是太棒了!")
- 永远积极向上
- 刻意的感叹号

**改用**:
- 客观陈述事实
- 允许中性或负面评价
- 真实的困惑和疑问

示例对比:
```
❌ AI味: "令人激动的是,这个工具为我们带来了前所未有的便利!"
✅ 自然: "这工具确实省事,虽然有些功能还不太稳定。"
```

### 4. 加入具体细节

**AI文章缺乏**:
- 具体数字
- 个人经历
- 真实案例
- 环境细节

**必须包含**:
- 具体场景("上周调试代码时...")
- 精确数据("3次失败后...")
- 真实反应("当时我也懵了")

示例对比:
```
❌ AI味: "在实际工作中,我们经常遇到这类问题。"
✅ 自然: "上周Vivado跑时序分析,setup violation报了200多个,debug了两天才定位到是RGMII的虚拟时钟约束写错了。"
```

## 语言风格指南

### 口语化表达

- 用"搞定"而不是"完成"
- 用"弄明白"而不是"理解"
- 用"挺好"而不是"令人满意"
- 允许用"其实"、"反正"、"就是"这类口语词

### 允许不完美

- 可以用"..."表示停顿
- 可以用括号加注释(就像这样)
- 允许短句、碎句
- 可以用"不过"、"但是"开头

### 个性化表达

- 允许用"我觉得"、"我的经验是"
- 可以表达不确定("这块我也没完全搞清楚")
- 可以有观点("个人更推荐X方案")

## 针对技术文章的特殊建议

因为用户是FPGA工程师,写技术文章时:

### 1. 用实际项目经验说话

```
❌: "时序分析是FPGA设计的重要环节。"
✅: "时序分析这块,我在做RGMII接口时踩过坑。虚拟时钟不绑实际端口会导致约束失效,这问题Vivado报告里不明显,得仔细看UG903才能发现。"
```

### 2. 说明工具的真实使用体验

```
❌: "Vivado提供了强大的时序分析功能。"
✅: "Vivado的时序报告挺详细,但看起来费劲。setup violation几百条的时候,得先看Histogram找出worst path,不然根本不知道从哪改起。"
```

### 3. 包含代码片段和具体数值

```
❌: "约束文件需要正确配置。"
✅: "RGMII的输入延迟约束得这么写:
set_input_delay -max 2.0 -clock [get_clocks RGMII_RXC] [get_ports rgmii_rxd[*]]
max值设2.0ns是按芯片datasheet算的,实测下来setup margin能留0.5ns左右。"
```

## 最终检查清单

写完文章后,检查以下几点:

- [ ] 没有"首先...其次...最后"的固定模式
- [ ] 删除了所有"在当今"、"综上所述"类的废话
- [ ] 至少有2-3个具体例子或数据
- [ ] 有真实的个人经验或观点
- [ ] 语言像是在跟同行聊天,不是在写论文
- [ ] 没有刻意的感叹号和假热情
- [ ] 段落长度不一,有长有短
- [ ] 允许有不确定的表达

## 核心记住一点

**写文章的目标是"说清楚事",不是"展示我会写文章"。**

把注意力放在具体内容上,让结构和修辞自然形成,别刻意追求"完美"。
